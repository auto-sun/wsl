# Mock 数据说明

## 当前原则

- 所有复杂能力先做框架和接口，不伪装成真实在线系统
- 页面可演示，但必须明确哪些数据来自 mock
- 所有 mock 数据结构尽量贴近未来真实业务结构

## 各模块说明

### Dashboard

- 首页概览卡片、图表、告警、地图点位来自后端 mock 数据
- 摄像头模块调用浏览器 `getUserMedia`，这部分是真实浏览器能力，不依赖后端

### Monitoring

- 地块、趋势、NDVI、巡检记录、热力图点位为 mock 数据
- 已预留未来接入无人机航测、遥感识别、多光谱分析的字段与说明

### Diagnosis

- 上传图片会真实写入 `media/diagnosis/`
- 会真实创建检测任务记录
- 推理结果由 `backend/services/ai/inference.py` 生成，当前 `inference_mode = "mock"`

### Decision

- 策略列表、处方等级、历史记录、地图多边形均为 mock 数据
- “生成新策略”会走后端接口与数据库记录，但结果仍为 mock 决策输出
- “下发策略”仅生成下发预览，不会触发真实灌溉施肥控制

### Devices

- 设备在线状态、日志、电量、信号、动作反馈均为 mock 数据
- 当前未接真实硬件、未接真实 MQTT Broker、未接 Redis 实时缓存

## 为什么这样设计

- 便于第一阶段先完成监测、诊断、决策、设备展示四条主流程
- 便于后续逐模块替换为真实服务，而不用推翻前端页面或 API 结构
- 便于竞赛/汇报场景演示完整业务链路
